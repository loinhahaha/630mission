# 功能评估报告

## 1) 当前工程是否满足“要实现的功能”

### 1.1 文档上传、自动获取文档内容
- **已基本支持**：后端 `/analyze` 支持 `mode=file` 上传文件；pipeline 会把 `.doc/.docx` 统一成 docx，再提取段落文本。  
- **边界**：`.doc` 依赖 LibreOffice，若环境无 `soffice` 会失败。

### 1.2 简单规则类解析
- **2.1 连续符号检测：部分支持**  
  已实现连续中文标点、中文/英文标点混用、标点前后空格等；但你列出的符号集合更广（如《》、引号组合、连字符等）还未完整覆盖。
- **2.2 特殊引号模式：未实现**  
  当前仅做“中文引号数量配对”，没有实现“`"内容1”"内容2"` 唯一合法、且中间不能有符号”的定制规则。
- **2.3 段末标点规则：未实现**  
  当前没有“每段必须以 `。/！` 结尾（含特例‘各位***、各位领导：’）”的规则。
- **2.4 标题上下空一行：部分支持（仅输出重排）**  
  在“生成 final.docx”时会在识别到标题后插入 1 个空段，但不是对“原文是否满足该规则”的检查。
- **2.5 段内不应出现空格：未实现**  
  目前只检查“中文标点前后空格”，没有做“段内任意空格（标题上下空行除外）”规则。

### 1.3 复杂规则类解析
- **3.1 行距 29 磅：未实现检查，也未设置输出行距 29 磅**。
- **3.2 脚注：未实现检查**。
- **3.3 页边距：部分支持**  
  有上/左边距检查（37/28mm）和输出模板设置（上37 左28 右26 下35）。
- **3.4 半角/全角统一：部分支持**  
  仅检测中英文标点紧邻混用，未做到“符号全部全角”的全量约束。
- **3.5 标题居中：部分支持**  
  生成 final.docx 时标题会居中，但未检查原稿标题是否居中。
- **3.6 正文首行缩进 2 字符：部分支持**  
  生成 final.docx 时统一设置首行缩进约 2 字；对原稿仅有非常弱的启发式提示。
- **3.7 字体字号分级（标题/一二三四级/正文）：未完整实现**  
  当前只对“识别为标题”的第一行设置小标宋近似二号，其余段落统一仿宋 16pt，未按层级做完整映射。

### 1.4 问题清单、行号/页码、标记后文档下载
- **问题清单：已支持**（`report.json` + UI 列表）。
- **问题对应行文字：已部分支持**（`evidence` 和段号 `paragraph_index`）。
- **问题所在页码：未支持**（python-docx 无稳定分页结果）。
- **标记后文档下载：已支持但标记样式有限**  
  可下载 `annotated_original.docx`，当前通过“插入红色说明行”标记，不是对原文 run 做“标黄/下划线/中划线”精细定位。

> 结论：当前工程可作为 **MVP 骨架**，但距离你列出的完整规则集仍有较大差距，尤其是“段末规则、特殊引号规则、脚注/行距/分级字体、页码定位、精细高亮标注”。

## 3) python-docx 的边界能力，是否可以实现目标功能

### 可以较好实现的部分
- 读取段落文本、段落级格式（对齐、缩进、段前后、行距）检查与写入。
- 页面设置（纸张、页边距）读取与写入。
- 字体字号、加粗/下划线/颜色等 run 级样式设置。
- 生成“标注版文档”（例如插入问题说明段、给特定 run 加高亮等）。

### 难点 / 边界
- **页码定位困难**：Word 分页依赖渲染引擎，python-docx 不能可靠告诉“某段在第几页”。通常要借助 Word COM（Windows）或 LibreOffice/商业引擎二次计算。
- **精确字符级回写标注复杂**：如果要在原文中精确“标黄某个字符区间”，需要维护 run 切分与偏移映射，工程复杂度高。
- **复杂版式语义识别有限**：如“一级/二级标题”更多依赖样式体系或规则+NLP，python-docx 只提供结构与属性，不提供语义判断。
- **字体可用性依赖系统**：你指定的方正字库若系统未安装，最终显示会回退。

### 是否能实现你的目标
- **能实现 70%~85%（工程可行）**：上传解析、规则检测、问题清单、标注下载、绝大部分格式规则都可用 python-docx + 规则引擎完成。
- **不能仅靠 python-docx 完美实现的关键点**：稳定页码、与 Word 完全一致的排版判定、复杂脚注语义和高保真回写。  
  建议方案：python-docx 负责主流程，页码与高保真定位引入 Word COM（Windows 专用）或 LibreOffice/PDF 中间层增强。

## 4) 你提出的两个问题与修复说明（已落地）

### 问题1：每段都标黄，只希望问题位置标注
- **原因**：原实现按“段落级”标注；只要某段存在标点问题，就把该段所有 run 全部设为黄色。
- **修复**：改为“按问题证据（evidence）定位字符区间”后再标注：
  - 标点/引号/全半角/括号：仅对命中的字符做黄色高亮；
  - 空格问题：仅对空格字符做中划线；
  - 格式类问题：优先仅下划线 evidence；若无可定位 evidence，才整段下划线（表示段级格式问题）。
- **结果**：不会再出现“只要有一个问题就整段全黄”的现象。

### 问题2：问题列表太长，希望可滚轮查看
- **原因**：原实现在展开区中直接顺序渲染，页面整体会被拉很长。
- **修复**：将“问题列表”放入固定高度容器 `st.container(height=420)`，内容超出时可在容器内滚轮滚动。

### 为什么经常看到“0个错误”？错误/警告/提示怎么区分
- **历史原因**：此前多数规则都被归类为 warning/info，error 几乎不产生。
- **现在的分级（已调整）**：
  - **error（错误）**：硬性规范不满足（如段末标点、段内空格、半角标点、标题上下空行、标题未居中、部分关键标点违规）；
  - **warning（警告）**：高风险但有上下文弹性的规范问题（如页边距偏差、称谓冒号等）；
  - **info（提示）**：优化建议或弱约束（如行距/缩进/字号建议、脚注提醒等）。
- **UI 已增加说明文案**：当错误=0时，也会提示“可能仍有警告/提示”。
